# 区域产业调研与机会决策系统现状与目标差距清单

> 对照基线：需求规格说明书 2.0、系统设计方案 2.0  
> 评估对象：当前 React 单区域原型  
> 用途：后续开发拆分、排期、验收和风险控制

## 1. 评估结论

当前系统已经具备可运行的单区域前端原型，完成了企业名单、统一企业详情、产业地图、问题生成、调研计划、录音上传、基础需求记录、政策静态匹配和图表展示。

但现阶段仍是“多个功能页面的集合”，尚未形成目标设计要求的业务系统。主要差距集中在：

1. 菜单按技术功能排列，没有按区域认知、调研执行、分析洞察和决策支持组织。
2. 企业、政策、情报、需求和结论的数据来源与可信状态不完整。
3. 研究专题与调研计划、企业样本、问题、证据和结论尚未形成闭环。
4. 政策可维护、版本化和重新匹配；动态情报已可人工维护、关联和核验，但尚未接入自动采集。
5. “需求归纳”和“建议结论”主要由简单汇总规则生成，缺少人工确认、证据、反例和审核过程。
6. 当前数据保存在浏览器本地，适合原型验证，不支持团队协作和可靠的数据治理。

因此后续应采用“保留可用组件、重组业务入口、补齐数据语义、再接入自动化”的方式演进。

## 2. 处理方式定义

| 处理方式 | 含义 |
| --- | --- |
| 保留 | 功能和业务定位基本正确，只需少量调整 |
| 重组 | 功能可复用，但需要移动到新的业务模块并改变上下文 |
| 升级 | 当前功能可用，但数据结构、状态和操作流程不足 |
| 替换 | 当前业务语义不正确，需要用新流程替代 |
| 新增 | 目标设计要求但当前不存在 |
| 延后 | 首期暂不实施，但设计和数据结构需预留 |

## 3. 菜单与信息架构差距

| 当前菜单 | 当前作用 | 目标归属 | 处理方式 | 主要差距 |
| --- | --- | --- | --- | --- |
| 区域概览 | 展示区域说明和数量统计 | 区域工作台 | 升级 | 缺近期动态、待办、样本覆盖、专题进度和阶段结论 |
| 数据看板 | 独立展示企业、计划、记录和需求图表 | 合并至区域工作台 | 重组 | 看板与业务行动脱节，且未明确统计口径 |
| 产业研究 | 手工维护专题和假设 | 分析洞察/研究专题 | 替换 | 专题缺研究问题、样本、证据、反例、结论和决策影响 |
| 企业名单 | 企业查询、导入、详情和编辑 | 区域认知/企业库 | 保留并升级 | 缺来源、核实状态、企业关系、动态时间线和需求证据 |
| 产业链地图 | 产业分类和核心节点网络 | 区域认知/产业全景 | 保留并升级 | 分类规则硬编码，关系缺证据，无法叠加政策、情报和调研结论 |
| 问题生成 | 按企业类型和产业位置生成问题 | 调研工作/调研准备包 | 重组 | 缺研究问题、假设、情报、政策和历史访谈等生成依据 |
| 政策匹配 | 静态政策规则匹配企业 | 区域认知/政策库 + 分析洞察/政策机会 | 替换 | 政策硬编码，无原文、版本、有效期、更新和重新计算 |
| 调研计划 | 单企业简单计划 | 调研工作/计划与样本 | 升级 | 缺多企业计划、样本理由、覆盖检查、假设和预期输出 |
| 调研记录 | 录音、纪要、需求和结论 | 调研工作/记录与行动项 | 升级 | 缺证据定位、原话/判断区分、需求确认、阻塞项和跟进任务 |
| 需求归纳 | 汇总记录中的需求 | 分析洞察/需求洞察 | 替换 | 无确认状态、证据、样本、反例、可信度和失效机制 |
| 建议结论 | 规则生成建议和区域结论 | 决策中心 | 替换 | 无洞察引用、审核发布、版本和分类决策输出 |
| 无 | 能力、案例和问题模板散落 | 知识资产 | 新增 | 缺统一资产管理、作用范围和版本 |

### 3.1 目标一级菜单

1. 区域工作台
2. 区域认知
3. 调研工作
4. 分析洞察
5. 决策中心
6. 知识资产

第一轮代码调整应只重组导航与页面归属，不立即删除现有组件和数据。

## 5. 数据架构差距

### 5.1 已有数据

- Workspace：已具备最小区域定义和 `workspaceId`。
- Company：已有基础画像、企业类型和产业链位置。
- ResearchPlan：已有企业、日期、负责人、目标和状态。
- ResearchRecord：已有企业、摘要、转写文本、需求、结论和录音名称。
- ResearchTopic/Hypothesis：已有最小专题和假设原型。
- Capability/QuestionTemplate：已有简单能力和问题模板。

### 5.2 缺失或不足的数据对象

| 数据对象 | 当前状态 | 必要改造 |
| --- | --- | --- |
| CompanySource | 缺失 | 保存企业信息来源、采集时间和核实状态 |
| CompanyRelation | 仅运行时推断 | 持久化关系类型、双方企业、证据和可信度 |
| Policy/PolicyVersion/PolicyRule | 静态常量 | 政策主记录、历史版本、有效期和匹配条件 |
| PolicyMatch | 运行时结果 | 保存匹配版本、条件、解释和重算状态 |
| IntelligenceItem | 已有原型 | 已保存招投标、采购、新闻、扩产、招聘、认证等情报及关联、核验状态；尚缺采集任务和统一证据链 |
| ResearchQuestion | 被专题和问题模板混用 | 独立定义本轮研究需要回答的问题 |
| ResearchSample | 缺失 | 保存选样企业、样本角色和选择理由 |
| InterviewPack | 缺失 | 保存每家企业的准备包和版本 |
| InterviewQuestion | 只有字符串 | 保存分类、顺序和生成依据 |
| Evidence | 缺失 | 统一引用录音片段、纪要、政策和情报 |
| NeedStatement | 嵌套在记录中 | 独立保存需求类型、状态、确认人和证据 |
| ActionItem | 缺失 | 保存负责人、截止时间和状态 |
| Insight | 缺失 | 保存共性判断、可信度、样本和审核状态 |
| DecisionRecommendation | 缺失 | 保存决策类别、依据、优先级和发布状态 |

### 5.3 当前最危险的数据语义

- 已移除招投标模拟卡片；外部线索均需录入来源、日期、企业/专题关联与核验状态。
- 需求保存后直接进入统计，没有人工确认状态。
- 产业关系由名称和关键词临时计算，没有来源和证据。
- 政策匹配基于硬编码关键词，没有政策版本和适用条件核实。
- 自动生成的结论没有区分系统建议与人工发布结论。

这些问题应在新增自动抓取和 AI 能力之前解决。

## 8. 保留、暂停和替换决策

### 8.1 继续保留

- 阎良企业基础数据和工作空间归属。
- 企业统一详情入口。
- 产业分类与核心节点协同两种视图。
- Excel 企业导入。
- 录音上传和调研纪要基础能力。
- 问题模板和能力库的基础数据。

### 8.2 暂停扩展

在数据语义和业务流程完成前，暂不继续扩展：

- 自动招投标抓取。
- 自动政策匹配规模化。
- AI 自动生成区域结论。
- 复杂机会评分。
- 多区域管理界面。

### 8.3 优先替换

- 独立“数据看板”替换为区域工作台驾驶舱。
- “问题生成”替换为企业调研准备包。
- “需求归纳”替换为有证据的需求洞察。
- “建议结论”替换为决策中心和审核发布流程。
- 静态“政策匹配”拆分为政策库和政策机会分析。

## 9. 建议开发阶段

### 阶段 A：信息架构与代码底座重整（P0）

- 按六个业务域重组导航。
- 将现有组件归位，暂不删除数据。
- 拆分 App.tsx，建立领域类型和数据访问层。
- 为 localStorage 增加数据版本和迁移测试。

验收：用户能明确区分事实数据、调研工作、分析洞察和决策输出；旧数据可正常迁移。

### 阶段 B：事实数据层（P0）

- 升级企业档案与来源。
- 持久化企业关系。
- 建设政策库、政策版本和政策规则。
- 建设动态情报及人工确认。（基础维护、关联企业/专题、核验状态和调研问题上下文已完成；自动采集待后续接入。）

验收：所有企业、政策和情报信息能说明来源、时间和验证状态。

### 阶段 C：调研闭环（P0）

- 研究问题与样本选择。
- 多企业调研计划和准备包。
- 访谈记录、证据、需求候选、确认和行动项。

验收：一轮调研可以从研究问题完整流转到经确认的企业需求和后续行动。

### 阶段 D：分析与决策（P1）

- 研究专题与证据链。
- 需求聚类、样本覆盖、反例和可信度。
- 市场、销售、研发、方案和生态建议。
- 区域研究简报。

验收：每条共性洞察和决策建议均可追溯到样本和证据。

### 阶段 E：自动化、团队化与多区域（P1/P2）

- 政策和情报采集任务。
- 语音转写和 AI 辅助。
- 服务端、权限、审计和对象存储。
- 多区域切换和全局管理看板。

## 10. 首个开发迭代建议

首个迭代只做阶段 A，不同时开发政策抓取或 AI：

1. 建立新的分组导航和路由结构。
2. 将区域概览与数据看板合并为区域工作台。
3. 将企业名单和产业地图归入区域认知。
4. 将问题生成、计划和记录归入调研工作。
5. 将产业研究、需求归纳归入分析洞察。
6. 将建议结论和能力库分别归入决策中心、知识资产。
7. 建立数据版本迁移和核心操作自动测试。

该迭代完成后，用户操作路径应变为：

`区域工作台 → 区域认知 → 调研工作 → 分析洞察 → 决策中心`。

## 11. 完成判定

- 差距项均能对应需求规格编号和开发阶段。
- 开发时不再按照旧菜单逐页增加功能。
- 新增功能必须明确所属业务域、数据来源、状态和下游用途。
- 每个阶段完成后执行构建、数据迁移测试和关键操作回归。
