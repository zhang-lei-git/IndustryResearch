# 区域产业调研与机会决策系统需求规格说明书

> 文档版本：2.2
>
> 首期试点：西安市阎良区航空制造产业
>
> 产品定位：区域产业研究工作台、调研助手与决策外脑

## 1. 建设目的

系统服务于数字化解决方案团队对产业聚集区域开展持续研究。它不以销售现有 MES、LIMS、MRO 或管理系统为起点，而是先理解区域产业运行、企业真实诉求和政府引导方向，再判断团队能够提供什么服务、需要补充什么能力、应采取什么市场策略。

系统需要支持完整业务闭环：

```mermaid
flowchart LR
  A[区域认知] --> B[提出研究问题]
  B --> C[选择调研对象]
  C --> D[编制访谈提纲与制定计划]
  D --> E[现场调研与证据沉淀]
  E --> F[需求确认与跨企业归纳]
  F --> G[政策和外部情报联合分析]
  G --> H[市场/销售/研发决策]
  H --> A
```

## 2. 产品边界与业务原则

### 2.1 系统负责

- 建立区域、产业、企业、政策和外部动态的结构化认知。
- 帮助用户明确每轮调研要回答的问题，而不是只维护企业名单。
- 基于企业角色、研究问题和已有证据准备调研计划与问题包。
- 保存录音、转写、纪要、企业明确诉求、研究人员判断和行动项。
- 将企业个案归纳为待验证主题、共性需求、反例和阶段性结论。
- 输出市场进入、销售策略、产品研发、行业方案和生态合作建议。

### 2.2 系统不负责

- 不把外部新闻、招聘或招标线索直接认定为企业需求。
- 不根据既有产品自动制造销售机会。
- 不替代 CRM、合同、回款和项目实施管理系统。
- AI 不直接发布结论，所有重要洞察必须可追溯并经人工确认。

### 2.3 数据可信原则

系统必须区分：客观事实、企业原话、外部线索、研究人员判断、AI 推断、已确认结论。任何推断都不得覆盖原始证据。

## 3. 用户与使用场景

首期为单人使用，未来扩展为团队协作。

| 角色 | 核心任务 |
| --- | --- |
| 调研负责人 | 建立区域研究框架、确定专题和样本、审核结论 |
| 调研执行人员 | 准备访谈、记录过程、提取需求、维护行动项 |
| 行业/方案专家 | 复核产业判断、政策适配和方案方向 |
| 市场与销售管理者 | 查看区域机会、销售策略和重点企业群 |
| 产品与研发管理者 | 判断共性需求、能力空白和产品投入方向 |

## 4. 信息架构与菜单

### 4.1 区域工作台

回答“当前区域研究到什么程度、最近发生了什么、下一步做什么”。

- 区域概况与产业主线
- 当前调研进度和样本覆盖
- 近期政策、招投标和企业动态
- 重点研究问题与待验证判断
- 待办事项和阶段性结论

### 4.2 区域认知

用于收集和查看事实数据。

- 产业全景：产业分类、核心节点协同、企业关系
- 企业库：企业画像、产业角色、关联企业
- 政策库：政策原文、版本、有效期、适用条件
- 动态情报：招投标、采购、新闻、扩产、招聘、认证等

### 4.3 调研工作

调研工作按“调研项目—调研对象—访谈提纲—调研计划—调研执行—调研记录库”组织，用于完成一次调研的准备、执行、查询和收尾。

- 调研项目：定义本轮调研目标、周期、负责人和关联分析专题，是后续对象、提纲、计划和记录的归属边界。
- 调研对象：在企业库按名称、行业、企业类型、产业链位置、规模和动态情报等条件筛选；筛选结果不会自动加入，只有用户明确加入后才成为“本轮要调研的企业”。
- 访谈提纲：以“调研项目 + 调研对象”为唯一编制上下文。从问题库选择、调整、排序和新增问题后形成；自定义或修改后的问题可人工同步回问题库。
- 调研计划：一次计划可安排多家企业；每家企业分别安排时间、方式、执行人和访谈提纲。
- 调研执行：只录入待执行企业安排的录音、纪要、需求和结论。
- 调研记录库：只查询已完成记录，不展示待执行计划；支持按调研项目或调研对象归集，并可打开单条完整记录详情。

界面不得直接使用“样本”“冻结”“计划对象”等数据模型术语；它们仅用于内部实现和接口定义。

### 4.4 分析洞察

用于把零散数据转化为有证据的区域认知。

- 研究专题
- 需求洞察
- 企业群和产业环节对比
- 政策与产业机会分析
- 证据、反例和不确定项

### 4.5 决策中心

- 市场进入建议
- 销售策略与优先企业群
- 产品和研发方向
- 行业解决方案建议
- 生态合作与政策包装建议
- 区域研究简报

### 4.6 知识资产

平台级共享资产，包括能力库、案例库、问题模板、行业模型、方案资产和数据源配置。

## 5. 核心业务需求

### BR-01 区域工作空间

- 阎良为第一个区域工作空间，后续可新增其他区域。
- 企业、调研、情报、区域洞察必须归属区域。
- 国家/省级政策、能力库、案例库和通用模板可跨区域复用。
- 区域驾驶舱只显示当前区域；未来另设全部区域管理看板。

### BR-02 产业全景

- 支持产业分类视图和核心节点协同网络两种视图。
- 产业关系不能只用“上游/下游”，还应包括设计输入、制造协同、质量要求、试验验证、问题反馈、供应商配套和数据协同。
- 企业、政策、动态、调研结论和需求主题可作为产业地图的叠加信息。
- 任意企业入口打开同一企业档案。

### BR-03 企业统一档案

企业档案应包括：

- 基础画像、产业角色、企业关系和信息来源
- 调研计划、历史访谈、录音、纪要和行动项
- 经人工确认的企业需求和待验证判断
- 招投标、新闻、扩产、招聘、认证等外部情报
- 政策匹配结果及匹配依据
- 相关研究专题和区域洞察

### BR-04 研究专题

研究专题代表一个需要通过多企业调研回答的业务问题，而不是普通分类目录。

- 用户可主动创建、修改和关闭专题。
- 系统可根据产业结构、政策变化、外部情报或重复出现的企业问题建议专题。
- 专题包含研究目的、假设、目标企业群、样本、问题、证据、反例、结论和决策影响。
- 专题状态包括：草稿、调研中、证据不足、形成阶段结论、已关闭。

### BR-05 调研任务、对象选样与样本

- 调研任务定义区域、产业范围、研究专题、调研目标、周期、负责人和预期输出，是后续对象、问题、计划和记录的归属边界。
- 可在企业库按企业类型、产业链位置、规模、链主协同关系、专题、近期情报、政策匹配、调研状态和调研价值等条件筛选。
- 筛选结果仅是企业库查询结果，不产生候选样本，也不自动写入调研项目。
- 用户从筛选结果中人工加入、移除、调整优先级并填写入选原因；加入后才形成调研对象。
- 样本角色至少包括链主、科研院所、典型配套、材料工艺、试验验证、平台服务、对照样本；系统提示链条或样本类型覆盖偏差。
- 企业主数据不等于调研对象。同一企业可进入不同任务，也可在同一任务中通过不同计划开展多次调研。

### BR-06 调研问题组与准备包

- 访谈提纲是可版本化、可复用、可编辑的调研问题集合，不等同于问题库条目或访谈记录；每份提纲必须归属一个调研项目和一个调研对象。
- 一个调研对象可对应多组提纲，例如首访画像、扩产专题、质量证据链专题和复访验证。
- 用户可基于项目、企业画像、关联专题、公开情报和政策点击“推荐问题”；推荐问题只作为待选择建议，必须人工加入提纲。
- 问题组保存调研重点、适用条件、分类问题、生成依据、来源情报/政策/专题、版本和冻结时间。
- 调研计划引用问题组时必须保存冻结快照；之后模板或问题组修改不得影响已执行计划。
- 系统为每家企业生成可编辑的准备包：

- 企业基础信息与产业位置
- 近期动态和招投标线索
- 适配政策及需要核实的条件
- 关联专题和待验证假设
- 分类调研问题及生成依据
- 既有案例只作为参考，不作为预设答案

### BR-07 调研计划、计划对象与执行

- 调研计划是时间和资源编排，可包含多个已加入项目的调研对象，不等同于企业清单。
- 一个计划对象唯一对应“调研任务样本 + 问题组快照 + 计划时间 + 执行人员”。同一计划内不得重复安排同一任务样本。
- 计划对象保存计划时间、预计时长、地点或方式、负责人、参与人、目标、关联情报、政策和状态。
- 支持按日历、时间线、列表和状态看板调整计划；计划时间与实际调研时间必须分别保存。
- 若同一企业需要再次调研，应创建新的计划对象并选择新的问题组或问题组新版本，不覆盖既有记录。

### BR-08 调研记录与证据

- 支持录音上传、手工纪要，后续支持语音转写。
- 每条调研记录必须且只能对应一个计划对象；记录自动继承企业、任务、专题、问题组和计划上下文。
- 调研记录库支持按调研项目或调研对象查看多次访谈，并在详情中查看录音、转写、摘要、结论、需求候选和访谈提纲快照。
- 区分企业原话、调研人员判断和系统提取结果。
- 支持记录需求、预算窗口、决策人、阻塞项、政策诉求、项目时间和下一步行动。
- 每项结构化内容应能够回到原始纪要、转写片段或外部材料。

### BR-09 需求确认与归纳

需求数据只能来自以下路径：

1. 企业访谈明确表达并由用户确认。
2. 企业正式文件、招标/采购文件等可靠材料，经用户确认。
3. 研究人员根据多条证据形成判断，并明确标记为“分析判断”。

需求状态包括：待确认、企业已确认、证据支持、存在反例、已失效。

系统按照企业类型、产业位置、区域、专题和时间聚类，但共性需求必须展示样本数量、证据来源、反例和可信度。

### BR-10 政策库与动态匹配

- 政策支持手工录入、文件导入、官方来源定期采集和手动检查更新。
- 政策需保存发布机构、层级、原文、版本、发布时间、有效期、申报窗口、适用对象、条件、支持方式和来源链接。
- 政策更新保留历史版本，不直接覆盖。
- 企业画像、调研结论或政策版本变化后，相关匹配标记为“需重新计算”。
- 匹配结果说明已满足条件、待确认条件、潜在支持和降低客户决策门槛的方式。

### BR-11 动态情报

- 支持人工维护、Excel/文件导入、手动刷新和定时采集。
- 每条情报保存来源、发布时间、采集时间、关联企业/专题、可信等级和验证状态。
- 情报只能触发“待验证信号、推荐问题或待办”，不能自动形成企业需求。

### BR-12 调研档案查询、洞察与决策

调研工作支持以下组合查询：企业名称、企业类型、产业链位置、规模、调研任务、专题、样本角色、问题组、计划状态、负责人、计划时间、实际调研时间、记录状态、需求确认状态、情报类型、政策匹配、是否有录音/转写、是否需要复访和证据完整度。

系统形成“研究问题—假设—证据—反例—结论—建议”的完整链路，并输出：

- 区域和产业环节的共性问题
- 市场进入与企业群优先级建议
- 销售切入场景和客户沟通策略
- 产品、研发和行业方案方向
- 政策杠杆和生态合作建议
- 可引用证据的阶段性研究简报

## 6. 看板范围

| 看板 | 数据范围 | 使用者 | 主要内容 |
| --- | --- | --- | --- |
| 区域驾驶舱 | 当前区域 | 调研团队 | 产业、企业、调研、情报、专题、洞察和待办 |
| 专题看板 | 当前专题 | 专题负责人 | 样本、假设、证据、反例、结论 |
| 全局管理看板 | 所有区域 | 管理层，后期 | 区域对比、共性需求、资源投入和能力复用 |

## 7. 数据来源与责任

| 数据类型 | 来源 | 默认状态 | 进入结论前要求 |
| --- | --- | --- | --- |
| 企业基础信息 | 公开资料、Excel、人工维护 | 待核实/已核实 | 标注来源和更新时间 |
| 企业原话 | 录音、转写、纪要 | 原始证据 | 不允许被自动改写覆盖 |
| 外部情报 | 官方网站、招采平台、新闻等 | 待验证 | 用户确认关联性 |
| 政策 | 政府官方来源、文件导入 | 待解析/有效 | 确认版本和有效期 |
| 企业需求 | 访谈或正式材料 | 待确认 | 人工确认并关联证据 |
| 共性洞察 | 多企业需求和证据 | 草稿 | 显示样本、反例和可信度 |
| 决策建议 | 洞察、能力和案例 | 草稿 | 负责人审核发布 |

## 8. 非功能需求

- PC 支持深度研究，手机支持企业查看、现场记录和待办处理。
- 所有区域业务数据具有 `workspaceId`，平台资产具有共享范围。
- 外部信息、政策和 AI 结果必须保留来源、版本和时间。
- 重要状态变化、人工确认和结论发布应可审计。
- 自动采集遵循公开来源规则和授权边界。
- 首期数百企业、数千记录保持常规操作流畅。

## 9. 验收标准

1. 用户能够从区域认知开始，完成计划、准备、记录、需求确认和阶段性结论。
2. 用户能明确看出每个模块的数据来源、当前区域和数据状态。
3. 企业档案在所有入口保持一致。
4. 外部情报不会未经确认直接成为企业需求。
5. 共性需求可以追溯到企业、调研记录和具体证据。
6. 政策支持更新、版本管理、重新匹配和匹配理由解释。
7. 系统能够新增第二个区域而不混淆区域原始数据和平台共享资产。
